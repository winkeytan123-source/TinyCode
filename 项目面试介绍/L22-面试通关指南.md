### Q3: 搜索替换编辑（edit_file）的工作原理？

**标准答案**：

> "edit_file 的核心是唯一匹配策略。LLM 提供三个参数：file_path、old_string、new_string。
>
> 执行逻辑：
> 1. 读取文件内容
> 2. 用 content.count(old_string) 检查出现次数
> 3. 如果出现 0 次，返回错误 + 文件开头预览
> 4. 如果出现超过 1 次，要求 LLM 提供更多上下文行
> 5. 如果恰好 1 次，执行替换并生成 unified diff
>
> 这种设计的优势是：
> - 无歧义：old_string 唯一确定位置
> - 可审查：diff 让用户看到具体改了什么
> - 容错性：如果匹配失败，LLM 可以根据反馈调整"

---

### Q4: 三层上下文压缩的触发条件和策略？

**标准答案**：

> "三层压缩基于 token 占用比例触发：
>
> **Layer 1 - 工具输出截断**（50% 阈值）：
> - 遍历所有 role='tool' 的消息
> - 如果 content 超过 1500 字符且超过 6 行，保留头 3 行 + 尾 3 行
> - 中间插入 '... (X lines, snipped to save context) ...'
>
> **Layer 2 - LLM 摘要**（70% 阈值）：
> - 保留最近 8 条消息，旧消息送给 LLM 总结
> - 总结 prompt：'压缩对话，保留文件路径、关键决策、错误信息、任务状态'
> - 将总结插入为新的 user/assistant 对，告知 Agent 这是压缩后的上下文
>
> **Layer 3 - 硬压缩**（90% 阈值）：
> - 只保留最后 4 条消息 + 总结
> - 这是最后手段，避免超出模型窗口限制
>
> token 估算用 `len(text) // 3`，对中英文混合场景有 ~3.5 chars/token。"

---

### Q5: 并行工具执行是怎么实现的？

**标准答案**：

> "当 LLM 返回多个 tool_calls 时，用 concurrent.futures.ThreadPoolExecutor 并行执行：
>
> ```python
> with ThreadPoolExecutor(max_workers=8) as pool:
>     futures = [pool.submit(self._exec_tool, tc) for tc in tool_calls]
>     return [f.result() for f in futures]
> ```
>
> 关键点：
> 1. 每个工具调用提交到线程池，立即返回 Future 对象
> 2. f.result() 会阻塞等待对应任务完成
> 3. 适合 read_file、bash、grep 这类 I/O 密集型操作
>
> Claude Code 的 StreamingToolExecutor 更激进——在 LLM 还没生成完全部 tool_calls 时就开始执行。CoreCoder 简化为：等 LLM 生成完所有 tool_calls，然后并行执行。"

### Q6: 危险命令拦截的实现？

**标准答案**：

> "bash 工具在执行前用正则匹配 9 类危险模式：
>
> ```python
> _DANGEROUS_PATTERNS = [
>     (r'\brm\s+(-\w*)?-r\w*\s+(/|~|\$HOME)', '递归删除 home/root'),
>     (r'\brm\s+(-\w*)?-rf\s', '强制递归删除'),
>     (r'\bmkfs\b', '格式化文件系统'),
>     (r':\(\)\s*\{.*:\|:.*\}', 'fork bomb'),
>     ...
> ]
> ```
>
> 如果匹配到任何一个模式，返回：
> '⚠ Blocked: {reason}\nCommand: {command}\nIf intentional, modify the command to be more specific.'
>
> 这不是 100% 的安全保证，但可以拦截大部分常见误操作。更安全的做法是沙盒执行（Docker 容器），但那会增加复杂度。"

### Q7: Agent 循环的核心逻辑？

**标准答案**：

> "Agent 的核心是一个 while 循环（最多 max_rounds=50 轮）：
>
> 1. 将 user 消息追加到 messages
> 2. 调用 llm.chat(messages, tools)，LLM 返回 text 或 tool_calls
> 3. 如果返回 text → 追加到 messages，返回给用户
> 4. 如果返回 tool_calls → 执行工具，将结果追加为 role='tool' 消息
> 5. 检查上下文是否需要压缩（context.maybe_compress）
> 6. 回到步骤 2
>
> 这个模式叫 ReAct（Reasoning + Acting）：LLM 推理出需要什么工具 → 执行工具 → LLM 基于结果继续推理。循环直到 LLM 认为任务完成（返回纯文本）。"

---

### Q8: 流式响应的实现细节？

**标准答案**：

> "流式响应通过 OpenAI SDK 的 stream=True 实现：
>
> ```python
> stream = self.client.chat.completions.create(
>     model=self.model,
>     messages=messages,
>     tools=tools,
>     stream=True,
> )
> ```
>
> 然后遍历 stream，逐 chunk 处理：
> - delta.content：文本增量，调用 on_token(delta.content) 实时打印
> - delta.tool_calls：工具调用增量，按 index 分组累积（id 传一次，name 传一次，arguments 分片传）
> - chunk.usage：最后一个 chunk 带 token 统计
>
> 关键难点是 tool_calls 的累积：
> - 一个 tool_call 可能拆成多个 chunk 传输
> - 用 dict[index] 按 index 分组，拼接 arguments 字符串
> - 全部接收完后 json.loads(arguments) 解析为 dict"

### Q9: 子代理（AgentTool）的实现？

**标准答案**：

> "AgentTool 让主 Agent 可以生成一个独立的子 Agent 处理复杂子任务：
>
> ```python
> sub = Agent(
>     llm=parent.llm,
>     tools=[t for t in parent.tools if t.name != 'agent'],  # 不允许递归
>     max_context_tokens=parent.context.max_tokens,
>     max_rounds=20,
> )
> result = sub.chat(task)
> ```
>
> 设计要点：
> 1. 子 Agent 共享 parent 的 llm，但有独立的 messages 和 context
> 2. 子 Agent 的工具列表排除 agent 工具本身，避免无限递归
> 3. 子 Agent 运行到完成（max_rounds=20），返回文本结果
> 4. 结果截断到 5000 字符，避免撑爆 parent 的上下文
>
> 适用场景：代码库调研、多步骤独立任务、需要「思考空间」的复杂推理。"

### Q12: 重试机制的实现？

**标准答案**：

> "LLM 调用用指数退避（exponential backoff）处理瞬时错误：
>
> ```python
> for attempt in range(max_retries):
>     try:
>         return self.client.chat.completions.create(**params)
>     except (RateLimitError, APITimeoutError, APIConnectionError) as e:
>         if attempt == max_retries - 1:
>             raise
>         wait = 2 ** attempt  # 1s, 2s, 4s
>         time.sleep(wait)
>     except APIError as e:
>         if e.status_code >= 500:  # 服务器错误，重试
>             time.sleep(2 ** attempt)
>         else:  # 客户端错误（4xx），不重试
>             raise
> ```
>
> 重试策略：
> - Rate Limit / Timeout / Connection Error：一定重试
> - 5xx 服务器错误：重试
> - 4xx 客户端错误（如参数错误）：不重试，直接抛出
>
> 等待时间呈指数增长：1秒 → 2秒 → 4秒，避免持续轰炸 API。"

### Q13: cd 命令追踪的实现？

**标准答案**：

> "bash 工具维护一个全局变量 _cwd 追踪当前工作目录：
>
> ```python
> _cwd: str | None = None
>
> def _update_cwd(command: str, current_cwd: str):
>     global _cwd
>     parts = command.split('&&')
>     for part in parts:
>         if part.strip().startswith('cd '):
>             target = part[3:].strip().strip('\"'\"')
>             new_dir = os.path.normpath(
>                 os.path.join(current_cwd, os.path.expanduser(target))
>             )
>             if os.path.isdir(new_dir):
>                 _cwd = new_dir
> ```
>
> 关键点：
> 1. 解析 `cd xxx` 或 `cmd1 && cd xxx && cmd2` 的情况
> 2. expanduser 处理 `~`，join 处理相对路径，normpath 规范化
> 3. 检查目录是否存在，存在才更新 _cwd
> 4. 下次执行命令时，用 _cwd 作为 subprocess 的 cwd 参数
>
> 这样 Agent 就能像真人一样「记住」当前在哪个目录，不需要每次都写绝对路径。"

### Q14: 系统提示词（system prompt）的设计？

**标准答案**：

> "系统提示词是 Agent 的「行为准则」，包含：
>
> 1. **身份定位**：'You are CoreCoder, an AI coding assistant...'
> 2. **环境信息**：工作目录、OS、Python 版本（动态注入）
> 3. **工具列表**：遍历 tools，格式化为 Markdown 列表
> 4. **行为规则**：8 条核心规则
>    - Read before edit（先读后改）
>    - edit_file for small changes（小改用 edit_file）
>    - Verify your work（改完验证）
>    - Be concise（简洁至上）
>    - One step at a time（一次一步）
>    - edit_file uniqueness（保证唯一匹配）
>    - Respect existing style（遵守现有风格）
>    - Ask when unsure（不确定时询问）
>
> 这个 prompt 是 Agent 行为的「宪法」，决定了它的工作风格。"

---

### Q15: unified diff 生成的实现？

**标准答案**：

> "edit_file 执行替换后，调用 difflib.unified_diff 生成 diff：
>
> ```python
> old_lines = old.splitlines(keepends=True)  # 保留换行符
> new_lines = new.splitlines(keepends=True)
> diff = difflib.unified_diff(
>     old_lines, new_lines,
>     fromfile=f'a/{filename}',
>     tofile=f'b/{filename}',
>     n=context,  # 上下文行数，默认 3
> )
> result = ''.join(diff)
> ```
>
> 关键点：
> - keepends=True 保留换行符，difflib 需要按行比较
> - a/ 和 b/ 前缀是 Git 约定，表示原始版本和新版本
> - n=context 控制显示多少行上下文，帮助定位变更位置
> - 超过 3000 字符的 diff 会截断，避免输出过长
>
> 生成的 diff 格式和 git diff 一样，用户可以直接看到改了什么。"

---

### Q16: Token 估算的实现？

**标准答案**：

> "用简单启发式估算 token 数：
>
> ```python
> def _approx_tokens(text: str) -> int:
>     return len(text) // 3
> ```
>
> 这基于经验值：中英文混合内容约 ~3.5 chars/token。实际的 tiktoken 编码会更准确，但需要额外依赖。
>
> 估算足够用于触发上下文压缩：
> - 50% 阈值触发工具截断
> - 70% 阈值触发 LLM 摘要
> - 90% 阈值触发硬压缩
>
> 真实 token 数从 LLM API 返回的 usage 字段获取，用于成本统计。"

---

### Q17: 为什么不用 LangChain 或 LlamaIndex？

**标准答案**：

> "CoreCoder 的目标是教学和理解，不是快速开发。如果用 LangChain：
> - 引入了额外的抽象层（Chain、Agent、Memory 等）
> - 隐藏了实现细节——你不知道 Agent 循环到底怎么工作的
> - 代码行数会膨胀（LangChain 自己都 20 万行+）
>
> CoreCoder 的核心理念是「可读性 > 功能完整性」：
> - 每个模块都是原生 Python，没有魔法
> - Agent 循环就是一个 122 行的 agent.py
> - 工具系统就是继承 Tool 基类的 7 个文件
>
> 类比：LangChain 是 PyTorch，CoreCoder 是 nanoGPT。前者是生产工具，后者是学习材料。"

### Q18: 项目的局限性？

**标准答案**：

> "主要局限：
> 1. **功能简化**：没有 Skills、MCP、多模态、语音模式等高级特性
> 2. **安全性**：危险命令拦截只是正则匹配，不是沙盒隔离
> 3. **性能**：单线程 Agent 循环，不支持真正的并发（多个用户同时使用）
> 4. **错误处理**：比较基础，没有 Claude Code 那样详细的错误分类和恢复策略
> 5. **测试覆盖**：测试用例不完整

> 但这些局限是有意为之——CoreCoder 是教学项目，不是生产工具。目标是让开发者理解核心架构，而不是提供完整的商业产品。"

### Q19: 如果让你改进这个项目，会做什么？

## 四、深挖追问应对策略

### 场景 1：为什么选择搜索替换而不是 diff patch？

**A（行动）**：
- 分析了三种编辑方案的优劣：整文件替换（浪费 token）、行号 patch（容易因上下文变化失效）、搜索替换（精确 + 容错）
- 搜索替换的核心是「唯一匹配约束」——old_string 必须在文件中恰好出现一次
- 如果匹配失败，返回详细错误（找不到 or 出现多次），LLM 可以根据反馈调整

**R（结果）**：搜索替换在 Claude Code 实践中被证明是最鲁棒的方案。CoreCoder 的实现只用 85 行代码，但捕获了核心思想。

---

### 场景 2：三层压缩会不会丢失重要信息？

**A（行动）**：
- Layer 1（工具截断）只压缩 verbose 输出（如长命令结果），保留头尾关键信息
- Layer 2（LLM 摘要）会指示 LLM「保留文件路径、错误信息、关键决策」，丢弃冗余内容
- Layer 3（硬压缩）是最后手段，只在 90% 窗口占用时触发
- 实际使用中，大多数对话只会触发 Layer 1，Layer 2/3 很少启动

**R（结果）**：压缩不可避免会有信息损失，但通过分层策略和 LLM 摘要，保留了最关键的上下文。如果任务确实需要长期记忆，应该用外部存储（如向量数据库）。

---

### 场景 3：并行执行工具时如何处理依赖？

**S（情境）**：LLM 有时会返回有依赖关系的工具调用，比如先 read_file 再基于内容 edit_file。如果并行执行，edit_file 可能在 read_file 之前完成，导致错误。

**T（任务）**：确保并行执行不会破坏工具之间的逻辑依赖。

**A（行动）**：
- CoreCoder 的策略是：信任 LLM 的调度能力
- 如果 LLM 需要 A 的结果来执行 B，它应该先返回 tool_call(A)，等结果后再返回 tool_call(B)
- 如果 LLM 在同一轮返回 A 和 B，说明它认为这两个操作是独立的（如同时读两个文件）
- 并行只在「同一轮返回的多个 tool_calls」生效，跨轮的调用天然串行

**R（结果）**：这个策略在实践中工作良好。Claude 3.5+ 的模型很少在同一轮返回有依赖的工具调用。如果遇到问题，可以在系统提示词中添加「One step at a time」规则。
