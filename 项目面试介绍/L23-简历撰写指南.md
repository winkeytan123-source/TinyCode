# L23 - CoreCoder 项目简历撰写指南

> 学习目标：掌握如何将 CoreCoder 项目写进简历，让面试官一眼看到你的技术深度。

---

## 前置说明

L22 已经涵盖了面试话术和 Top 20 面试题。本节聚焦于**简历文字本身**——如何用最少的字数，传递最大的信息密度，让你的简历在筛选阶段就脱颖而出。

简历和面试是两个战场：**简历决定你能不能进门，面试决定你能不能拿 Offer。** 很多人技术不差，但简历写得像流水账，连面试机会都拿不到。

## 一、简历项目描述的核心原则

### 1.1 STAR 法则在简历中的应用

STAR 法则是结构化描述项目经历的黄金框架。但在简历中，四个要素的篇幅分配需要调整——**Action 占大头，Result 紧随其后**，Situation 和 Task 用半句话带过即可。

| 要素 | 含义 | 简历中的体现 | CoreCoder 对应示例 |
|------|------|-------------|------------------|
| **S**（Situation） | 项目背景 | 一句话交代为什么做 | Claude Code 源码泄露 51 万行，难以理解 |
| **T**（Task） | 你的目标 | 融入项目标题或第一行 | 逆向分析并提炼核心架构 |
| **A**（Action） | 你做了什么 | 每个 bullet point 的核心 | 实现 7 个核心模式、1400 行 Python... |
| **R**（Result） | 取得了什么成果 | 量化数据收尾 | ~1400 行代码、7 个模式、支持 100+ 模型 |

**反面教材：** Situation 写了一大段"随着 AI Agent 技术的飞速发展……"——面试官没时间看你写作文。

**正确做法：** 把背景压缩到项目标题或第一行：

```
CoreCoder —— 从 51万行到 1400行的 AI Agent 架构提炼
```

一行就同时交代了 S（51 万行难以理解）、T（架构提炼）、R（1400 行）。

### 1.2 简历描述的"三要三不要"

#### 三要

**1. 要量化成果**

没有数字的简历等于没有锚点。

| 维度 | 量化示例 |
|------|---------|
| 代码量对比 | 51 万行 TypeScript → ~1400 行 Python |
| 架构模式 | 7 个核心设计模式 |
| 压缩层数 | 3 层上下文压缩策略 |
| 支持模型 | 20+ 主流模型定价支持 |
| 文章影响力 | 7 篇技术导读、16 万字、17 万+ 阅读 |
| 工具数量 | 7 个核心工具 |

**2. 要突出技术深度**

| 模糊描述 | 精准描述 |
|---------|---------|
| 实现了 Agent 系统 | 实现了 ReAct 模式 Agent 循环（LLM 推理 → 工具执行 → 结果反馈） |
| 做了上下文管理 | 三层上下文压缩（50%截断 / 70%LLM摘要 / 90%硬压缩） |
| 支持多模型 | OpenAI 兼容 API + LiteLLM (100+ 模型提供商路由) |
| 实现了文件编辑 | 搜索替换编辑（唯一子串匹配 + unified diff） |
| 做了安全防护 | 正则匹配 9 类危险命令（rm -rf、fork bomb、chmod 777） |

**3. 要体现工程能力**

```
✓ 流式响应：逐 token 实时输出，delta.content 增量累积，tool_calls 按 index 分组拼接
✓ 指数退避重试：Rate Limit / Timeout / 5xx 错误自动重试（1s / 2s / 4s）
✓ ThreadPool 并行：多工具调用并发执行，submit + result 阻塞收集
✗ "实现了一些优化"（这几个字等于什么都没说）
```

#### 三不要

**1. 不要写"参与了""了解了"这种弱动词**

| 弱动词（扣分） | 强动词（加分） |
|---------------|---------------|
| 参与了逆向分析 | 独立完成 51 万行源码的逆向分析 |
| 了解了 Agent 架构 | 深度研究并提炼出 7 个核心架构模式 |
| 学习了上下文压缩 | 实现了三层上下文压缩策略 |
| 使用了并行执行 | 基于 ThreadPoolExecutor 实现工具并行调度（max_workers=8） |

**2. 不要堆砌不相关的技术栈**

只写和项目直接相关的技术。CoreCoder 是 Python CLI Agent，不要出现 React、MySQL、Redis。

**3. 不要夸大——面试会拷打每一个字**

简历上写了"三层压缩"，面试官就会问"三层分别是什么？触发条件？"。写了"并行执行"，就会问"怎么处理依赖？"。写了"危险命令拦截"，就会问"拦截了哪些模式？"。

**铁律：简历上的每个技术名词，你都要能展开讲 2 分钟。**

## 三、普通写法 vs 优化写法对比

### 对比 1：项目背景描述

**普通写法：**
> 研究了 Claude Code 的源码并做了一个类似的项目。

**面试官内心：** "多少行代码？研究了什么？类似到什么程度？"

**优化写法：**
> 逆向分析 Claude Code 泄露源码（512,664 行 TypeScript），提炼出 Agent 循环、工具系统、上下文压缩等 7 个核心架构模式，用 ~1400 行 Python 重构实现。

**面试官内心：** "这人有逆向分析大型代码库的能力，知道怎么提炼核心模式，可以深挖。"

---

### 对比 2：编辑工具描述

**普通写法：**
> 实现了文件编辑功能。

**面试官内心：** "什么编辑？怎么实现的？和 vim 有什么区别？"

**优化写法：**
> 实现搜索替换编辑（Claude Code 的 FileEditTool）：LLM 指定 exact substring（必须在文件中唯一出现），执行替换后生成 unified diff。避免了整文件传输和行号 patch 的脆弱性。

**面试官内心：** "理解了设计权衡，知道为什么这样做，准备问问唯一匹配的实现细节。"

---

### 对比 3：上下文管理描述

**普通写法：**
> 做了上下文压缩优化。

**面试官内心：** "什么压缩？怎么压的？触发条件是什么？"

**优化写法：**
> 实现三层上下文压缩策略（参考 Claude Code 的 HISTORY_SNIP / Microcompact / CONTEXT_COLLAPSE）：
> - Layer 1（50% 阈值）：截断 role='tool' 消息的 verbose 输出，保留头尾各 3 行
> - Layer 2（70% 阈值）：LLM 摘要旧消息（保留最近 8 条），压缩为 user/assistant 对
> - Layer 3（90% 阈值）：硬压缩只保留最后 4 条消息 + 摘要

**面试官内心：** "三层策略很清晰，每层的触发条件和实现逻辑都说了，这是深入研究过的。"

---

### 对比 4：并行执行描述

**普通写法：**
> 支持多个工具同时执行。

**面试官内心：** "用什么并发模型？asyncio 还是线程？如何处理依赖？"

**优化写法：**
> 基于 ThreadPoolExecutor 实现工具并行执行（max_workers=8）：当 LLM 同一轮返回多个 tool_calls 时，
> 用 submit 提交到线程池，f.result() 阻塞收集结果。模拟 Claude Code 的 StreamingToolExecutor
>（原版 530 行，CoreCoder 简化为 agent.py 中的 _exec_tools_parallel 方法）。

**面试官内心：** "知道 ThreadPool 的原理，理解原版和简化版的差异，可以聊聊为什么不用 asyncio。"

---

### 对比 5：安全机制描述

**普通写法：**
> 加入了一些安全检查。

**面试官内心：** "什么检查？检查了什么？怎么检查的？"

**优化写法：**
> 危险命令拦截（参考 Claude Code 的 BashTool 1143 行实现）：正则匹配 9 类危险模式——
> rm -rf（递归删除 home/root）、chmod 777 /（root 目录全开权限）、fork bomb（: (){ :|:& };:）、
> pipe curl to bash（curl xx | bash）等。匹配到则返回警告并阻止执行。

**面试官内心：** "知道具体拦截了哪些模式，理解这不是 100% 安全但是实用的权衡。"

---

### 对比 6：多模型支持描述

**普通写法：**
> 支持多种 LLM 模型。

**面试官内心：** "哪些模型？怎么支持的？切换方便吗？"

**优化写法：**
> 支持任意 OpenAI 兼容 API（通过 base_url + api_key 切换：Kimi、DeepSeek、Qwen、Claude via OpenRouter、Ollama）。
> 对于不兼容的提供商（AWS Bedrock、Google Vertex），通过 LiteLLM 后端路由到 100+ 模型提供商。
> 硬编码 20+ 主流模型定价表（input/output per million tokens），用于 Token 统计和成本估算。

**面试官内心：** "架构设计考虑了扩展性，知道不同提供商的接口差异，还做了成本追踪。"

## 六、简历避坑指南

### 6.1 绝对不要写的内容

**❌ "跟着教程做了一遍"**

即使你确实是读了文章后做的，也不要这样写。

正确做法：
```
✗ 跟着 CoreCoder 教程完成了 AI Agent 实现
✓ 基于 CoreCoder 架构独立实现了 XXX 功能/优化
✓ 深入研究了 CoreCoder 的上下文压缩策略并做了对比实验
```

**❌ 写了但说不清楚的技术**

如果你写了"三层压缩"但说不出三层分别是什么，面试官会认为你在简历造假。

**❌ 项目 Star 数（如果不是你的项目）**

"CoreCoder 获得 XXX Stars" —— 这是作者的成就，不是你的。

**❌ 复制粘贴 README**

面试官见过无数份简历，如果你的项目描述和 README 一模一样，他会直接判定"只是跑了代码"。

### 6.2 面试官最关注的点

**问题一：你在项目中的具体贡献是什么？**

- 如果你逆向分析了源码 → 写"逆向分析"
- 如果你基于 CoreCoder 做了扩展 → 写"基于 CoreCoder 架构实现了 XXX 功能"
- 如果你只是跑了代码 → 不要把这个项目写在简历上

**问题二：你对技术原理的理解深度如何？**

| 你写的内容 | 面试官的判断 |
|-----------|------------|
| "实现了 Agent 系统" | 入门级，可能只会调库 |
| "实现了 ReAct 模式 Agent 循环" | 中等，知道关键概念 |
| "实现了 ReAct 循环（Reasoning→Tool Call→Result→Loop），最多 50 轮" | 较深，有实际动手经验 |
| "实现了 ReAct 循环，并行执行工具时用 ThreadPool 而非 asyncio，因为工具大多是 subprocess 调用，GIL 影响小" | 深入，理解设计权衡 |

**问题三：你遇到了什么问题、怎么解决的？**

准备 2-3 个"踩坑-排查-解决"的小故事：

- 流式响应时 tool_calls 分片累积的处理

## 附录：简历常见 Q&A

**Q：我只是读了 CoreCoder 代码，能写"实现"吗？**

A：如果你完整读懂了每个模块的实现，可以写"深入研究了 CoreCoder 的 XXX 实现"。如果你基于它做了扩展或修改，可以写"基于 CoreCoder 架构实现了 XXX"。

**Q：我基于 CoreCoder 做了改进，怎么写？**

A：把你的改进作为单独的 bullet point：

```
• 在 CoreCoder 基础上优化了上下文压缩策略：
  - 实现了自适应阈值（根据对话长度动态调整 50%/70%/90%）
  - 加入了摘要质量评分（ROUGE-L），低质量摘要回退到截断
  - 对比实验：优化后 token 节省 15%，任务完成率提升 3%
```

**Q：CoreCoder 是开源项目，面试官会不会觉得"大家都能做"？**

A：同一个项目，不同人写出来的简历差距巨大。面试官看的是你的理解深度和表达方式。能讲清楚"为什么用搜索替换""三层压缩的触发逻辑"的人，永远不会和"跑了一遍代码"的人混淆。

**Q：简历上的技术栈行应该写什么？**

A：只写和项目直接相关的核心技术：

```
✓ Python / OpenAI API / ThreadPool / Regex / JSON
✗ Python / JavaScript / React / MySQL / Redis / Docker / ...
  （不要把不相关的技术栈堆上去）
```
