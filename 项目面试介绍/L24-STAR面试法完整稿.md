## 一、STAR 面试法介绍

### 1.1 什么是 STAR 面试法？

STAR 面试法是一种结构化的回答框架，用于在面试中清晰、有逻辑地展示个人经历和能力。四个字母分别代表：

| 要素 | 英文 | 含义 | 核心问题 |
|------|------|------|----------|
| **S** | Situation | 情境/背景 | 当时的环境和上下文是什么？ |
| **T** | Task | 任务/目标 | 你需要解决什么问题？目标是什么？ |
| **A** | Action | 行动/措施 | 你具体做了什么？用了什么方法？ |
| **R** | Result | 结果/成效 | 最终效果如何？你学到了什么？ |

**一句话总结**：先交代背景，再明确任务，然后详述行动，最后量化结果。

### 1.2 为什么 STAR 面试法有效？

1. **结构清晰**：面试官能快速抓住核心信息
2. **有说服力**：用具体事实而非抽象自评来证明能力
3. **防止跑题**：四步框架像导轨，让你不会失控
4. **便于追问**：面试官可以在任意一步深入
5. **适配技术面**：技术面试官喜欢"量化结果"和"工程细节"

### 1.3 使用 STAR 的注意事项

| 常见误区 | 正确做法 |
|----------|----------|
| S 和 T 讲太久 | S+T 控制在总回答的 20% 以内 |
| A 部分只说"我们做了" | 必须用"**我**"来描述自己的贡献 |
| R 部分含糊其辞 | 尽量量化：1400 行、7 个模式、3 层压缩 |
| 编造不存在的经历 | 只讲真正做过的事 |
| 每个 STAR 都讲 5 分钟 | 核心回答 1-2 分钟，追问再展开 |

**黄金比例建议**：

```
S（10%）→ T（10%）→ A（60%）→ R（20%）
```

### 3.2 分模块 STAR 话术

#### 3.2.1 搜索替换编辑模块

**S（情境）**：Agent 需要频繁编辑文件。传统方案有两个问题：整文件传输浪费大量 token（一个 1000 行的文件可能只改 2 行），行号 patch 容易因上下文变化失效（比如用户在文件开头插入了几行）。

**T（任务）**：需要一种既节省 token 又鲁棒的编辑方案。

**A（行动）**：
1. 分析 Claude Code 的 FileEditTool 实现（分散在多个文件中，总计约 400 行）
2. 理解其核心思想：用 exact substring match 定位编辑位置
3. 实现 85 行的 edit_file 工具：
   - LLM 提供 file_path、old_string、new_string
   - 用 content.count(old_string) 检查出现次数
   - 0 次 → 返回错误 + 文件开头预览
   - 多次 → 要求 LLM 提供更多上下文行
   - 恰好 1 次 → 执行替换，生成 unified diff
4. diff 生成用 difflib.unified_diff，保留头尾上下文各 3 行

**R（结果）**：edit_file 成为 CoreCoder 最常用的工具。唯一匹配约束消除了歧义，unified diff 让用户可以清楚看到改了什么。实践中 90%+ 的编辑都能一次成功，失败时 LLM 会根据错误信息调整 old_string。

#### 3.2.2 三层上下文压缩模块

**S（情境）**：Agent 长时间运行会产生大量消息历史——用户输入、LLM 回复、工具输出。模型窗口有限（如 128K tokens），当对话进行到第 10+ 轮时，上下文会逼近窗口限制，导致无法继续对话。

**T（任务）**：设计一个多层压缩策略，在保留关键信息的前提下，尽量延长 Agent 的运行时间。

**A（行动）**：
1. 研究 Claude Code 的 4 层压缩策略：HISTORY_SNIP、Microcompact、CONTEXT_COLLAPSE、Autocompact
2. 简化为 3 层实现（context.py 196 行）：
   - **Layer 1（50% 阈值）**：遍历 role='tool' 的消息，如果 content 超过 1500 字符且超过 6 行，保留头 3 行 + 尾 3 行，中间插入 "... (X lines, snipped) ..."
   - **Layer 2（70% 阈值）**：保留最近 8 条消息，旧消息送给 LLM 总结（prompt: "压缩对话，保留文件路径、关键决策、错误信息"）。总结插入为 user/assistant 对，告知 Agent 这是压缩后的上下文
   - **Layer 3（90% 阈值）**：只保留最后 4 条消息 + 总结，这是最后手段
3. Token 估算用简单启发式：len(text) // 3（约 3.5 chars/token）
4. 每次工具执行后自动检查是否需要压缩（context.maybe_compress）

**R（结果）**：三层压缩让 Agent 可以持续运行 30+ 轮对话而不崩溃。Layer 1 通常足以应对大多数情况，Layer 2/3 很少触发。压缩后的上下文仍然保留了最关键的信息——文件路径、错误记录、当前任务状态。

#### 3.2.3 并行工具执行模块

**S（情境）**：LLM 有时会同时需要多个工具的结果，比如同时读取 3 个文件、或者并行执行多个 grep 搜索。如果串行执行，每个工具都要等前一个完成，总耗时是各工具的累加。

**T（任务）**：当 LLM 一次返回多个 tool_calls 时，并行执行以提升效率。

**A（行动）**：
1. 分析 Claude Code 的 StreamingToolExecutor（530 行，非常复杂——它在 LLM 还没生成完全部 tool_calls 时就开始执行）
2. CoreCoder 简化为：等 LLM 生成完所有 tool_calls，然后并行执行
3. 用 concurrent.futures.ThreadPoolExecutor 实现（agent.py 中的 _exec_tools_parallel 方法）：
   ```python
   with ThreadPoolExecutor(max_workers=8) as pool:
       futures = [pool.submit(self._exec_tool, tc) for tc in tool_calls]
       return [f.result() for f in futures]
   ```
4. 选择 ThreadPool 而非 asyncio，因为工具执行大多是调用外部进程（subprocess），Python 的 GIL 在这种场景下影响小，ThreadPool 更简单

**R（结果）**：当 LLM 返回 3 个 read_file 调用时，并行执行耗时约等于最慢的那个文件读取，而不是 3 次读取的总和。对于独立的 I/O 操作，速度提升明显。

**T（任务）**：让主 Agent 可以生成一个独立的子 Agent 处理复杂子任务，子 Agent 运行完成后返回总结。

**A（行动）**：
1. 研究 Claude Code 的 AgentTool（1397 行，包含 Worktree 隔离、状态管理等复杂逻辑）
2. 提炼核心思想：子 Agent 是一个独立的 Agent 实例，有自己的 messages 和 context
3. 用 58 行实现 agent.py 中的 AgentTool：
   - 共享 parent.llm（不需要重新创建 LLM 客户端）
   - 独立的 messages 和 context（不污染 parent）

**R（结果）**：子代理让主 Agent 可以「委派」复杂任务。比如"调研这个代码库的模块划分"，子 Agent 可以读 10+ 个文件、执行多次 grep，最后返回一个总结报告给主 Agent。主 Agent 的上下文保持干净。

#### 3.2.5 危险命令拦截模块

**S（情境）**：Agent 执行 bash 命令有风险——LLM 可能生成 `rm -rf /` 或其他破坏性命令。虽然大多数情况下 LLM 不会故意这样做，但误操作或幻觉可能导致灾难。

**T（任务）**：在 bash 工具中加入安全检查，拦截高风险命令。

**A（行动）**：
1. 分析 Claude Code 的 BashTool（1143 行，包含复杂的安全检查和输出管理）
2. 提炼出 9 类危险模式并用正则匹配：
   - `rm -rf /` or `~` or `$HOME`（递归删除 home/root）
   - `chmod 777 /`（root 目录全开权限）
   - `: (){ :|:& };:`（fork bomb）
   - `curl xx | bash`（pipe curl to bash）
   - `mkfs`（格式化文件系统）
   - `dd ... of=/dev/`（原始磁盘写入）
   - 等
3. 每次执行命令前先调用 _check_dangerous(cmd)
4. 如果匹配到危险模式，返回警告并阻止执行：
   `⚠ Blocked: {reason}\nCommand: {command}\nIf intentional, modify to be more specific.`

**R（结果）**：危险命令拦截提供了基本的安全保障。虽然不是 100% 的防护（正则可以被绕过），但可以拦截大部分常见误操作。更安全的方案是 Docker 沙盒，但那会增加复杂度。

## 四、技术难点追问 STAR 应对

### 场景 1：流式响应中 tool_calls 的累积处理

**S（情境）**：OpenAI 的流式 API 会把一个 tool_call 拆成多个 chunk 传输。比如一个 tool_call 的 id、name、arguments 可能分散在 5 个 chunk 中。arguments 是 JSON 字符串，会被拆成多个片段。

**T（任务）**：正确累积这些片段，最后还原成完整的 tool_call 对象。

**A（行动）**：
1. 研究 OpenAI API 的流式响应格式：每个 chunk 包含 delta.tool_calls，每个 tool_call 有 index
2. 用 dict[index] 按 index 分组存储：
   ```python
   tc_map: dict[int, dict] = {}  # index -> {id, name, args}
   ```
3. 遍历 stream，处理每个 chunk：
   - delta.content：直接累积到 content_parts
   - delta.tool_calls：按 index 分组
     - id 只传一次，直接覆盖 tc_map[idx]['id']
     - name 也只传一次
     - arguments 是碎片，用 += 拼接
4. 全部接收完后，json.loads(args) 解析为 dict

**R（结果）**：正确处理了流式 tool_calls 的累积。即使一个 tool_call 被拆成 10 个 chunk，也能完整还原。关键是理解 index 的作用——它是分组的依据。

### 场景 2：cd 命令追踪的实现

**S（情境）**：Agent 执行 bash 命令时，用户可能会 cd 到某个目录。但每次 subprocess.run 都是独立的进程，不会保留工作目录。如果 Agent 执行了 `cd /tmp && ls`，下次执行 `ls` 时又回到原目录。

**T（任务）**：让 bash 工具能「记住」当前工作目录，就像真实的 shell 会话一样。

**A（行动）**：
1. 维护一个全局变量 _cwd 追踪当前工作目录
2. 每次执行命令时，用 _cwd 作为 subprocess.run 的 cwd 参数
3. 命令执行成功后（returncode == 0），调用 _update_cwd 解析命令中的 cd：
   - 用 split('&&') 拆分命令串联
   - 检查每个 part 是否以 `cd ` 开头
   - 提取目标目录，用 os.path.expanduser（处理 ~）、os.path.join（处理相对路径）、os.path.normpath（规范化）
   - 检查目录是否存在，存在则更新 _cwd
4. 测试边界 case：`cd ..`、`cd ~/project`、`cmd1 && cd /tmp && cmd2`

**R（结果）**：bash 工具现在能正确追踪 cd。Agent 可以执行 `cd src`，下次执行 `ls` 时会在 src 目录下。这让 Agent 的行为更像真人操作 shell。

### 场景 3：会话持久化的安全设计

**S（情境）**：会话持久化需要把 session_id 作为文件名。如果用户提供的 session_id 包含 `../` 或绝对路径，可能导致路径穿越攻击（读写任意文件）。

**T（任务）**：确保 session_id 安全化，所有会话文件都保存在 ~/.corecoder/sessions 目录内。

**A（行动）**：
1. 用正则清洗 session_id：
   ```python
   _SAFE_SESSION_RE = re.compile(r"[^A-Za-z0-9._-]+")
   name = _SAFE_SESSION_RE.sub("-", session_id).strip(".-_")
   ```
2. 构建路径后用 resolve() 获取绝对路径：
   ```python
   path = (SESSIONS_DIR / f"{name}.json").resolve()
   root = SESSIONS_DIR.resolve()
   ```
3. 检查 path 的父目录是否是 root：
   ```python
   if root != path.parent:
       raise ValueError("Invalid session id")
   ```
4. 只有通过检查的路径才能进行文件操作

**R（结果）**：会话持久化现在是安全的。即使用户提供 `../../etc/passwd` 作为 session_id，也会被清洗成合法的文件名或拒绝。所有会话文件都在 ~/.corecoder/sessions 目录内，不会泄露到其他位置。

### 场景 4：指数退避重试的实现

**S（情境）**：调用 LLM API 时可能遇到各种瞬时错误：Rate Limit（请求频率过高）、Timeout（请求超时）、503（服务器临时不可用）。这些错误是暂时的，如果立即抛出会导致 Agent 中断。

**T（任务）**：实现一个鲁棒的重试机制，对瞬时错误自动重试，同时避免无限重试。

**A（行动）**：
1. 识别可重试的错误类型：
   - RateLimitError：请求频率过高
   - APITimeoutError：请求超时
   - APIConnectionError：连接错误
   - APIError 且 status_code >= 500：服务器错误
2. 实现指数退避（exponential backoff）：
   ```python
   for attempt in range(max_retries):
       try:
           return self.client.chat.completions.create(**params)
       except (RateLimitError, APITimeoutError, APIConnectionError) as e:
           if attempt == max_retries - 1:
               raise
           wait = 2 ** attempt  # 1s, 2s, 4s
           time.sleep(wait)
   ```
3. 对于 4xx 客户端错误（如参数错误），不重试，直接抛出
4. max_retries 设为 3，最多等待 1 + 2 + 4 = 7 秒
5. 对于 LiteLLM 后端，用字符串模糊匹配错误类型（因为异常类不统一）

**R（结果）**：重试机制显著提升了 Agent 的鲁棒性。遇到临时的 Rate Limit 或网络抖动时，Agent 会自动等待并重试，而不是直接崩溃。等待时间呈指数增长，避免持续轰炸 API。

### 5.2 "你是如何做技术选型的？"

**S（情境）**：在实现并行工具执行时，我面临一个选择：用 ThreadPoolExecutor 还是 asyncio？

**T（任务）**：选择一个既简单又有效的并发方案。

**A（行动）**：
1. **分析工作负载**：工具执行大多是调用外部进程（subprocess.run）、文件 I/O（Path.read_text）、HTTP 请求（openai.chat.completions.create）。这些都是 I/O 密集型，不是 CPU 密集型。
2. **评估 asyncio**：
   - 优势：事件驱动，单线程高并发
   - 劣势：需要所有工具都是 async 函数，subprocess 需要用 asyncio.create_subprocess_exec，改动大
   - Python 的 GIL 在 I/O 密集型场景下影响小
3. **评估 ThreadPool**：
   - 优势：简单，现有代码改动小（只需 submit + result），支持任何可调用对象
   - 劣势：线程开销（但 8 个线程的开销可以接受）
4. **参考 Claude Code**：StreamingToolExecutor 用的是类似线程的并发模型（虽然 TypeScript 用的是 Promise + async/await）
5. **决策**：选择 ThreadPoolExecutor，因为简单性 > 理论性能优势

**R（结果）**：ThreadPoolExecutor 在实践中工作良好。对于同时读取 3 个文件这种场景，性能提升明显。代码保持简洁，没有因为 asyncio 而增加复杂度。这个选择验证了「简单性是终极复杂性」的原则。

**S（情境）**：在测试三层上下文压缩时，我发现 Layer 2（LLM 摘要）有时会失败——LLM 返回空字符串或者只有几个词，导致压缩后上下文丢失了大量信息。

**R（结果）**：优化后 Layer 2 的成功率从 60% 提升到 95%+。更清晰的 prompt 让 LLM 知道要保留什么信息，fallback 机制保证了即使 LLM 失败也有基本的信息提取。这次调试让我理解了「prompt 工程」在实际系统中的重要性。

## 七、面试模拟对话

> 以下是一段完整的模拟面试对话

---

**【第 1 轮：开场】**

**面试官**：你好，先简单介绍一下你做过的项目吧。

**候选人**：面试官好。我最近做了一个叫 CoreCoder 的项目，是对 Claude Code 51 万行源码的逆向分析和架构提炼。今年 3 月 Anthropic 的 npm 包泄露了完整源码，我花了两周深度阅读，提炼出 7 个核心架构模式——Agent 循环、搜索替换编辑、三层上下文压缩、并行工具执行等。然后用 1400 行 Python 重构了这些模式，实现了一个可运行的 AI Agent。如果您感兴趣，我可以展开介绍某个部分。

---

**【第 2 轮：技术深挖】**

**面试官**：那你说说搜索替换编辑具体怎么实现的？

**候选人**：搜索替换编辑是 Claude Code 的 FileEditTool 的核心设计。LLM 提供三个参数：file_path、old_string、new_string。

执行逻辑是这样的：首先读取文件内容，然后用 content.count(old_string) 检查出现次数。如果出现 0 次，返回错误加文件开头预览，让 LLM 知道文件内容是什么；如果出现超过 1 次，要求 LLM 提供更多上下文行来保证唯一性；如果恰好 1 次，执行替换并生成 unified diff。

这个设计的优势是消除了歧义——old_string 唯一确定了编辑位置，不需要行号，也不会因为文件其他部分的修改而失效。

**面试官**：如果 LLM 提供的 old_string 包含错别字怎么办？

**候选人**：那会匹配失败（count 返回 0）。工具会返回错误信息加文件开头预览，LLM 看到反馈后会修正 old_string。我在测试中发现，大多数情况下 LLM 第二次就能成功匹配。如果想更鲁棒，可以加入模糊匹配（difflib.SequenceMatcher），但那会增加歧义性，所以我保持了精确匹配。

---

**【第 3 轮：架构理解】**

**面试官**：你提到三层上下文压缩。为什么需要三层，不是一层或两层？

**候选人**：这是一个分层防御的设计思路。

第一层（50% 阈值）是最轻量的——只截断 verbose 的工具输出，比如 bash 命令返回了 5000 行日志，我们保留头尾各 3 行。这一层几乎无损，因为中间的大部分内容通常是重复或不重要的。

第二层（70% 阈值）是 LLM 驱动的摘要，这会有一定的信息损失，但能把 20 条旧消息压缩成 2 条。我们保留最近 8 条消息不动，只压缩更早的部分。

第三层（90% 阈值）是硬压缩，只保留最后 4 条消息加总结，这是最后手段——说明对话已经非常长了，必须大刀阔斧地压缩。

为什么需要三层？因为不同的压缩策略有不同的成本和损失。轻量压缩频繁触发但几乎无损，重量压缩偶尔触发但损失较大。分层设计让 Agent 尽可能延长运行时间，同时控制信息损失。

---

**【第 4 轮：工程能力】**

**面试官**：你实现了指数退避重试。为什么选择指数而不是固定等待时间？

**候选人**：指数退避是处理瞬时错误的标准实践。

如果用固定等待时间（比如每次都等 2 秒），在高负载场景下会导致「惊群效应」——所有失败的请求在 2 秒后同时重试，再次导致 Rate Limit，然后又同时等 2 秒...形成周期性的流量尖峰。

指数退避（1秒 → 2秒 → 4秒）让不同请求的重试时间自然地分散开，降低了再次碰撞的概率。而且等待时间递增，给了服务器更多的恢复时间。

max_retries 设为 3 是一个经验值——大多数瞬时错误在 7 秒内（1+2+4）会恢复，如果 3 次还失败，很可能是持续性错误（如 API key 错误），继续重试也没意义。
